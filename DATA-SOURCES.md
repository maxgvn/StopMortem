# What's real and what's fake in this demo

StopMortem mixes real, live API calls with entirely fabricated deal data. This page is the map — read it before treating anything the UI shows as fact.

## External integrations

| Integration | Status | Detail |
| --- | --- | --- |
| **Anthropic Claude API** | Real | Every Stage 1 and Stage 3 call is a genuine Messages API request. Nothing about the LLM output is mocked. |
| **Sillage** | Real | `lib/sillage/`. Live REST calls against `api.getsillage.com` with a real API key — **company-level** enrichment via the Top Account List, workspace signals, persona lookup. Sillage cannot resolve individual people; person-level lookups are FullEnrich's job (below). Medidata Solutions (`medidata.com`) and Box (`box.com`) were added to the **real, live** Top Account List on this account during development. The persona on that account was also genuinely expanded (via `PUT /persona`) to include IT-buyer and architect titles alongside the existing clinical-ops titles. **Caveats:** (1) `GET /persona` proved unreliable to read back on this account — it returned different, unrelated personas across consecutive calls (French insurance/fleet-management content unrelated to anything here), apparently a scoping issue on Sillage's side on the shared account this was built against; writes are more trustworthy than reads for that specific endpoint. (2) A real bug this caused: an early Medidata run's "Market signals" section included a paragraph about Box's company size, falsely attributed to Sillage — the model was citing an unrelated persona/signal result as if it concerned the deal at hand. Fixed in Stage 1's system prompt (`pipeline/stage1-evidence.js`) — see "Fixed issues" below. |
| **FullEnrich** | Real | `lib/fullenrich/`. Live REST calls against `app.fullenrich.com/api/v2` with a real API key — **person-level** company lookup, named-person lookup, company search, reverse-email lookup. Every enrichment result shown in the UI is genuine, current data (e.g. GitLab's real data correctly shows Sid Sijbrandij stepped down as CEO in Dec 2024 — the fixture below is stale on this point, and the *agent's own analysis* correctly caught and noted the discrepancy). **Caveat:** `people/search` (filter-by-seniority bulk search) has an undocumented request wire format that didn't respond to extensive testing — it is not wired in. Stakeholder resolution instead relies on `lookup_person` (name + company domain) and `reverse_email` (email → identity), both confirmed working. |
| **HubSpot / CRM** | **Fully fake** | Not integrated at all. Every "deal" is a local JSON fixture (`fixtures/deals/*.json`) standing in for a HubSpot export, including the `company.hubspot.accountTier` field ("strategic" vs. "standard") — a fabricated stand-in for a real HubSpot account-tier property. The "Simulate CRM auto-trigger" button / `POST /api/webhooks/deal-closed` demonstrates the *shape* of a real trigger — there is no real webhook receiver, signature verification, or HubSpot workflow behind it. `server.js` also fires this same simulated webhook automatically at startup for the first lost deal plus one other random lost deal, so the app never lands on an empty state. The "Published to → HubSpot (deal record)" line in every report is explicitly marked `simulated` in the output, not presented as a real write. See below for exactly what's fabricated in the deal content itself. |

## Fixture deals — real vs. invented, per deal

Every fixture pairs a **real company** (so live enrichment calls return genuine data) with an **entirely invented sales narrative** (deal amount, stage, notes, activity timeline, emails, proposal terms, closed-lost reason). The company is real; the deal never happened.

| Fixture | Real | Invented |
| --- | --- | --- |
| `deal-001` (Box) | Company: Box (`box.com`). One anchor stakeholder, Aaron Levie (real CEO, used only as an enrichment test target — never actually "in" the fabricated deal). | Deal amount, stage, all notes/activity/emails/proposal. The "IT Director" stakeholder is a title-only placeholder — no real name, by design (the gap the deal is built to demonstrate). |
| `deal-002` (Asana) | Company: Asana (`asana.com`). Anchor stakeholder: Dustin Moskovitz (real co-founder/CEO, same caveat as above). | Deal narrative, budget-conflict story, "VP Operations" title-only placeholder, named competitor Rippling (real company, fictional competitive mention). |

Named competitor Veeva Vault CTMS (real company). `meta.industry: "life_sciences_pharma"` — the only fixture that activates the pharma compliance add-on framework (see below). | `deal-005` (GitLab) | Company: GitLab (`gitlab.com`). Anchor stakeholder: Sid Sijbrandij (real founder — see the caveat above about his current real title being out of date in this fixture). | Deal narrative, budget-cut story, "Engineering Manager" title-only placeholder. |

The pipeline's evidence-gathering/gap-scan/synthesis stages are deliberately not run against them (`POST /deals/:id/run` returns a 400 for `stage: "closed_won"`) — a loss-analysis pipeline has nothing to analyze on a deal that closed clean, so this is an honest scope boundary, not a missing feature. |

**Why title-only placeholders matter:** stakeholders known only by title, with no name in the "CRM," are not a shortcut — they're the intended gap. Stage 1's job includes trying to resolve who actually holds that title via FullEnrich's `reverse_email` (using a synthetic email address seeded in the fixture's `emails[]`) or `lookup_person`. Whether it succeeds or returns "not found" is itself real signal about enrichment coverage, not scripted.

**The `deal-003-feedback.json` fixture** is entirely invented (a fabricated quote from "the VP Clinical Development") — it exists to demonstrate the feedback-upgrade mechanism (Stage 2, `pipeline/feedback.js`), not to represent a real client conversation. **Manual feedback entered through the UI's Stage 2 tab is genuinely whatever the person using the tool types** — it's not pre-scripted, but it's also not verified against anything; the tool trusts what's entered the same way the fixture does.

## The pharma/life-sciences compliance add-on is a real, standard framework — applied deterministically

`config/frameworks/pharma-compliance-addon.json` (21 CFR Part 11 electronic-records compliance, data residency/HDS-style hosting, IQ/OQ/PQ validation documentation, formal security sign-off) reflects genuine, standard regulatory concerns for clinical/life-sciences software — this isn't invented terminology. What *is* deliberately narrow: it only activates for `deal-003` (the one fixture with `meta.industry: "life_sciences_pharma"`) via a plain code check (`appliesToIndustry` in `pipeline/gap-scan.js`) — there's no real classification model deciding "is this deal pharma," just an explicit fixture field. Treat this as a demonstration that the framework layer is genuinely swappable/composable, not as a claim that StopMortem auto-detects industry.

## Publish targets — one real, three explicitly simulated

Every finished report lists where it was distributed (`postmortem.publishTargets`, `output/publish-targets.js`). Only **"Local file"** (`simulated: false`) reflects something that actually happened — the JSON/Markdown really is on disk at the given path. The other three ("HubSpot (deal record)", "Slack (#sales-team)", "Notion (recurring-causes log)") are `simulated: true` with a `detail` string describing what a real integration *would* do — no HubSpot note was written, no Slack message was posted, no Notion page was touched. The UI renders these with a hollow circle (○) vs. a checkmark (✓) specifically so this distinction is visible, not just documented here.

## Fixed issues worth knowing about (found via real runs, not hypothetical)

- **Redundant tool calls.** An early Medidata run's tool log showed `fullenrich_reverse_email` called 4 times with the identical email address, and a second email called twice — pure waste. Fixed with in-run memoization (`pipeline/stage1-evidence.js`'s `callKey`/`callCache`) — an identical tool+args call within one run now reuses the cached result.
- **Cross-company mislabeling.** A Datadog run's "Market signals" included a line reading like a restatement of the deal's own `identifyPain` field, falsely prefixed "Sillage:" as if it were an external signal. A separate Medidata run's market signals included analysis of Box's company size — unrelated to the deal at hand, likely bleeding in from Sillage's persona-read unreliability (see above). Both are now explicitly forbidden in Stage 1's system prompt: `marketSignals` entries must come from a real tool call made *this session*, and must concern the deal's own company or a company it actually names as a competitor — restating internal CRM fields as if they were external signals, or citing an unrelated company's profile, is disallowed.
- **Third-party review triggering regardless of evidence clarity.** `pipeline/next-step-classifier.js` originally checked the deal-size/pipeline-depth threshold *before* checking whether a finding was already confirmed and clear — so a big, deep-pipeline deal (e.g. Medidata at $90k/negotiation) got `full_third_party_review` on every single finding, even ones a client had already confirmed with high confidence. Fixed by checking "is this finding already a high-confidence documented_gap" first: a confirmed, clear finding now resolves to `no_further_investigation_needed` regardless of deal size, while genuinely unresolved findings on the same deal (or on a deal with no feedback at all, like Datadog) still correctly escalate.

## What's deterministic vs. LLM-generated

Independent of real-vs-fake data, it's worth knowing what's *computed* vs *written by Claude*:

- **Deterministic (code):** the numeric rubric score per finding (`pipeline/rubric-scoring.js`), the recommended next-step category (`pipeline/next-step-classifier.js`), which framework owns each finding (`tagFindingsWithFramework` in `pipeline/gap-scan.js`), the next-steps rollup (`buildNextStepsRollup` in `pipeline/stage3-scoring.js`), and the publish-targets list (`output/publish-targets.js`). Stage 3's Claude call receives the score and next-step values as fixed inputs and does not alter them (verified: Stage 3 output scores are byte-identical to Stage 1's computed scores in every run so far).
- **LLM-generated (Claude):** which gap findings exist at all, their evidence tier, their citations and confidence, the prose explanations, the proposed remedial actions, the department-specific takeaways, and the optional recovery note.
